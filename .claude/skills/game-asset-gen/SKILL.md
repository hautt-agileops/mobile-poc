# game-asset-gen

Worker-less pipeline: a game's `ASSETS.md` analysis → an `assets.manifest.json`
contract → real 2D PNG assets generated by Vertex AI nano-banana → dropped into a
Unity project's `Assets/Resources/Art/` so the runtime loads them. Zero npm deps
(pure Node 18+: `fetch`, `Buffer`, `crypto`). No server, no worker.

Reuses the 3d-prompt skill's credential resolution verbatim (`sa.json` / `GOOGLE_*`
env / 1Password), so the same Vertex service account works here.

## When to use

- A 2D game prototype (especially a unity-poc build) needs actual art — character
  sprites, stage tiles, UI icons, backgrounds, concept boards — not flat boxes.
- The `unity-poc` workflow reaches the asset step: it writes `ASSETS.md` +
  `assets.manifest.json`, then calls this skill to fill `Assets/Resources/Art/`.
- Standalone: any pipeline with an `assets.manifest.json` can call `gen-assets.mjs`.

## Workflow

1. **Analyze the design docs.** Read the game's `PRD.md` + `TDD.md` (and any
   `game-concept.md`). Extract every visual the prototype renders: each character
   (idle + key action frames), each stage/background, platform/tiles, UI (health
   bar, buttons, select portraits, logo), FX (hit spark, projectile), and 1–2
   concept boards that fix the palette and mood. Write this up as **`ASSETS.md`** —
   the human-readable analysis: a table of asset → type → size → **frames** → one-line
   art intent, plus a global style guide paragraph (palette, line weight, era/genre,
   lighting).

   **Detect animation intent.** If the design docs describe motion — a walk/run
   cycle, attack/cast, idle bob, hit/death, spawn, projectile spin, looping FX, or
   any verb implying the sprite moves over time — that asset is a **spritesheet**,
   not a single `sprite`. Look for words like *animate, cycle, loop, walk, run,
   attack, cast, idle, hit, die, spawn, wind-up, recover, frames, fps*. In `ASSETS.md`
   list its frame count and a per-frame beat breakdown. A static asset (tile, UI icon,
   background, portrait) stays `sprite`/`tile`/`ui`/`bg` with 1 frame. When the docs
   are silent on whether a character animates, default its core states (idle, plus
   the one signature action) to short spritesheets — a moving prototype reads better
   than a frozen one.

   **Frame-count guide (pixel-art):** idle 2–4 · walk/run 4–8 · attack/cast 3–5
   (wind-up → strike → recover) · hit 2 · death 4–6 · looping FX 3–6. Keep counts low:
   more frames = more model drift + more tokens, and pixel-art reads fine at 6–8.

2. **Author `assets.manifest.json`** — the machine contract `gen-assets.mjs` reads.
   One entry per asset; the global `style` string is prepended to every prompt so
   the set stays coherent. See `references/manifest-schema.md` for the full schema.
   Keep ids stable and Unity-friendly (`fighter_red_idle`) — they become the PNG
   filename and the `Resources.Load` key. Use `ref` to chain related assets off a
   generated one (e.g. every move frame refs the idle) for on-model consistency.

   **Animated assets → `type: "spritesheet"` + `frames` + `frameNotes`.** Each
   spritesheet is generated in ONE multi-turn call (prior frames fed back as context),
   so the character stays on-model across the cycle. Always give it `frameNotes` — one
   pose beat per frame — and `ref` the character's idle so palette/proportions lock.
   The generator writes `<id>_0.png … <id>_{N-1}.png`; the framework's
   `SpriteLoader.GetFrames(id, n)` loads them and game code flipbooks through them
   (or drop them on an Animator clip). Example — a run cycle refed off the idle:
   ```json
   {
     "id": "fighter_red_run", "type": "spritesheet", "category": "character",
     "prompt": "the same red brawler mid-run", "frames": 6, "ref": "fighter_red_idle",
     "frameNotes": ["contact L","recoil","passing L","contact R","recoil","passing R"],
     "background": "transparent"
   }
   ```
   List the idle (single `sprite`) BEFORE any spritesheet that refs it — array order
   is generation order.

3. **Dry-run the prompts** (cheap sanity check, generates nothing):
   ```bash
   node gen-assets.mjs <project>/assets.manifest.json -d
   ```
   Reads each composed prompt; tighten any that read wrong before spending tokens.

4. **Generate** — writes PNGs into the manifest's `outDir` (default
   `Assets/Resources/Art/`, relative to cwd, so run from the project root or pass
   `-o <project>/Assets/Resources/Art`):
   ```bash
   cd <project> && node <skill>/gen-assets.mjs assets.manifest.json
   ```
   Existing PNGs are skipped (idempotent) — re-run to fill only what failed. `-F`
   forces a full re-gen, `-i <substr>` targets a subset, `-o` overrides the dir.

5. **Key out the background (REQUIRED for `transparent` sprites).** nano-banana does NOT
   emit a real alpha channel — for a "transparent background" it *paints* a fake-
   transparency checkerboard as opaque RGB. Strip it to real alpha with the bundled keyer
   (border flood-fill; the dark subject outline stops the fill, interior lights survive):
   ```bash
   python3 <skill>/alpha_key.py -d <project>/Assets/Resources/Art --skip bg_cafe,concept_keyart
   ```
   Skip full-scene `bg`/`concept` assets (no cut-out wanted). Idempotent — files that
   already have alpha are left alone. Needs Pillow. Verify with
   `sips -g hasAlpha <file>.png`.

5. **Hand back to Unity.** PNGs now sit in `Assets/Resources/Art/<id>.png`. The
   unity-poc framework's `SpriteLoader.cs` loads them at runtime by id and falls
   back to `PrimitiveArt` flat-color for any id that didn't generate — so a partial
   or failed asset run still produces a playable build. Continue the unity-poc
   workflow at its playtest/build step.

## How assets reach the game

- **Location:** `Assets/Resources/Art/<id>.png`. `Resources.Load<Texture2D>("Art/<id>")`
  works synchronously in WebGL (no `UnityWebRequest`/StreamingAssets async dance).
- **Id == key:** the manifest `id` is the filename stem AND the lookup key. Game
  code asks `SpriteLoader.Get("fighter_red_idle")`.
- **Fallback is the safety net:** missing PNG → flat-color box. The headless build
  never breaks on a missing asset; it just degrades to programmer art for that one.
- **Spritesheets:** `frames > 1` writes `<id>_0.png … <id>_N.png`, generated in one
  multi-turn so the frames stay on-model. `SpriteLoader.GetFrames(id, n)` loads them.

## Provider / model defaults

- Image: `gemini-3-pro-image-preview`, remapped to **`gemini-3.1-flash-image`**
  ("Nano Banana 2") on Vertex — same override the 3d-prompt pipeline uses. gemini-3.x
  ids route to the Vertex `global` endpoint. Override with `GEMINI_IMAGE_MODEL`.
- Vertex AI only — no Gemini Developer API-key path.
- One image per asset (or N for a spritesheet). No text-refine pass — the manifest
  prompts are already concrete.
- **Parallel by default.** Assets generate concurrently (default 4 at a time, `-c N`
  or `GEN_CONCURRENCY` env to change). `ref`-dependent assets wait for their ref (a
  barrier between dependency levels keeps chains on-model); spritesheet frames stay
  serial (multi-turn). Per-request 5× exponential backoff on 429/503 absorbs Vertex
  throttling, so a full run finishes with 0 failures even when rate-limited.

## Credentials (auto-resolved — do NOT ask the user to export env vars first)

Identical resolution to the 3d-prompt skill, in order: `sa.json` next to this skill
or in cwd → `GOOGLE_APPLICATION_CREDENTIALS` → `GOOGLE_SA_OP_REF` (1Password) →
`GOOGLE_SERVICE_ACCOUNT_JSON` (inline, single-quoted) → `GOOGLE_VERTEX_ACCESS_TOKEN`.
Project auto-derives from the SA's `project_id`. Only ask for credentials if both the
service account file AND the 1Password pull are unavailable.

## Gotchas

- **Alpha varies by model.** Nano Banana 2 (`gemini-3.1-flash-image`) usually returns a
  real alpha channel already; NB1 (`gemini-2.5-flash-image`) returned an opaque RGB with a
  *painted* light checkerboard (or flat fill). Either way, ALWAYS run `alpha_key.py`
  (step 5) on `transparent` sprites before the game loads them — it's idempotent, a
  `cleared 0%` line just means the image was already keyed. Verify with `sips -g hasAlpha`.
- **The `ref` field over-reaches on inanimate props.** Referencing a character-rich
  concept board (e.g. `concept_keyart` with a mascot) when generating a plain object
  (`prop_plate`) makes the model inject that mascot into the prop. Drop `ref` for
  inanimate items and add hard negatives ("no character, no mascot, no person"). If it
  still contaminates, delete the PNG and let the `SpriteLoader` fallback cover it.
- **Run from the project root** (or pass `-o`): `outDir` is resolved relative to cwd,
  not to the manifest path.
- **Idempotent by filename:** an asset whose PNG exists is skipped. Delete the PNG or
  pass `-F` to regenerate. A failed asset leaves no PNG, so a plain re-run retries it.
- **Never** load the SA via `set -a; source` — bash strips the JSON's double quotes.
  Use a `sa.json` file or a single-quoted inline value (same as 3d-prompt).
- **`sa.json` is a real private key** — gitignored, never commit or print it.
- Keep `outDir` under `Assets/Resources/` — Unity only auto-includes `Resources/`
  folders in the build for `Resources.Load`. Elsewhere the PNGs won't ship.

## Run / develop

```bash
node gen-assets.mjs assets.manifest.json -d          # dry run: inspect prompts
node gen-assets.mjs assets.manifest.json             # generate all (skip existing)
node gen-assets.mjs assets.manifest.json -i fighter  # only ids containing "fighter"
node gen-assets.mjs assets.manifest.json -c 8        # 8 parallel (default 4)
node gen-assets.mjs assets.manifest.json -c 1        # force fully serial
node gen-assets.mjs assets.manifest.json -F          # force full re-gen
node gen-assets.mjs assets.manifest.json -o ./Art    # override output dir
```
